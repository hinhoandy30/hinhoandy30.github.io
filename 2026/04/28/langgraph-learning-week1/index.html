从 LLM 训练到 Agent 开发：我学 LangGraph 的第一周

做了一段时间 LLM 训练之后，我开始想搞清楚”用大模型做应用”到底是怎么一回事。选了 open_deep_research 这个项目作为切入点——它是一个能自动搜索、分析、输出研究报告的 Agent 系统，代码量不大，但把 LangGraph 的核心用法都覆盖到了。

这篇文章是第一周的学习记录，聊聊我踩的坑、搞懂的概念，以及一些让我豁然开朗的类比。

我的背景

熟悉 LLM 训练那一套——logits、beam search、gradient accumulation 这些概念都不陌生。但 Agent 开发、Python 工程生态、后端开发，之前完全没碰过。

所以这个学习过程其实很有意思：底层原理理解起来快，但工程惯例和框架用法需要从零开始建立直觉。

坑二：response_format type is unavailable

项目用了 with_structured_output 来强制模型按格式输出，但第三方代理 API 报这个错。

背后原因是结构化输出有两种底层策略：

  • ProviderStrategy：把 JSON Schema 直接发给 API，要求原生支持（OpenAI 官方 API 支持，很多代理不支持）
  • ToolStrategy：把 schema 包装成工具定义，走 function calling，兼容性更好

坑三：deepseek-reasoner does not support this tool_choice

我用的 deepseek-v4-flash，但 API 内部把它路由到了 R1（推理模型）。R1 不支持 tool calling，所以 Agent 直接挂了。

解法：换成 deepseek-chat（DeepSeek-V3）。

这里学到一个重要认知：推理模型（R1 类）不能用在需要 tool calling 的 Agent 场景。推理模型擅长深度思考，但 Agent 框架依赖工具调用来控制流程，两者目前不兼容。

搞懂 LangGraph vs LangChain：一直傻傻分不清楚

刚开始我总是混淆这两个，遇到报错不知道该查哪个文档。

后来总结出一句话区分它们：

LangGraph 管图，LangChain 管模型。

具体来说：

框架 负责的事 典型用法
LangGraph 图结构、节点、边、状态机、流程控制 StateGraph, add_node, add_edge
LangChain 模型调用、工具绑定、输出格式化 bind_tools, with_structured_output

看报错的 traceback 路径就能快速判断：langgraph/ 开头查 LangGraph 文档，langchain_core/ 开头查 LangChain 文档。

LangGraph 的三个核心概念

理解了这三个东西，整个框架就通了。

State：所有节点共享的”记忆”

State 是一个字典，贯穿整个执行流程，所有节点都从它读数据、往它写数据。

对我来说最好的类比是：State 就像神经网络里的 hidden state，每一层（节点）都在读取和更新它。

Annotated[list, operator.add] 这个写法的意思是：多个节点同时写这个字段时，用 operator.add 合并（列表追加），而不是互相覆盖。这个机制叫 reducer。

Node：处理单元

Node 就是普通的 Python 函数（或者子图），接收 State，返回要更新的字段：

有意思的地方是：普通函数可以作为节点，一个完整的子图也可以作为节点挂进父图——因为编译后的子图实现了完全相同的接口。

最绕的部分：约束解码是怎么工作的

with_structured_output 背后到底在做什么？这个问题我想了很久才真正搞清楚。

核心机制：有限状态机（FSM）+ logit masking

把 JSON Schema 转成一个有限状态机（FSM），FSM 的每个状态代表”当前 JSON 写到了哪里”。每次模型生成下一个 token 之前，FSM 会计算出哪些 token 是合法的，把其他所有 token 的 logit 设为 -∞——这样模型就只能生成符合格式的内容。

这和 beam search 里的路径屏蔽很像，只不过细化到了 token 级别。

一个很有意思的问题：FSM 怎么知道字符串内容什么时候写完了？

答案是：它不知道，也不管。当模型在写字符串内容时，”继续写”和”写结束引号”都在合法 token 集合里。什么时候结束，由模型自己的语言能力决定。FSM 只保证一件事：required 字段没写完之前，} 永远不在合法集合里。

这是”硬约束”和”模型智能”的完全分离——FSM 是纯硬代码算法，和模型权重无关，只负责格式合法性。

Pydantic BaseModel 有两种用途

在这个项目里，BaseModel 出现在两个完全不同的场景：

  1. structured output schema：定义模型必须输出的格式，with_structured_output(MySchema) 里用
  2. 工具定义：定义 bind_tools 里的工具，模型自己决定要不要调用

Field(description="...") 的内容会被塞进发给模型的 JSON schema，本质上是 prompt，直接影响模型的理解和输出质量。

为什么 State 用 TypedDict 而不是 BaseModel

因为 LangGraph 内部把 State 当字典处理，TypedDict 更轻量，不需要 Pydantic 的运行时验证。BaseModel 用在需要验证的地方（structured output），TypedDict 用在 State 定义。

关键概念速查表

概念 一句话说清楚
State 所有节点共享的字典，贯穿整个执行周期
Reducer 多节点并发写同一字段时的合并策略（追加 or 覆盖）
Node 处理单元，普通函数或子图均可
Edge 控制流，三种形式：固定 / 条件 / Command
Command(goto, update) 节点内部动态路由 + 更新 state
compile() 把 builder 冻结成可执行图对象
子图作为节点 编译后的子图实现相同接口，可直接挂进父图
bind_tools 给模型注册工具，模型自己决定调不调
with_structured_output 强迫模型输出符合 schema 的结构
ProviderStrategy JSON Schema 约束解码，需原生 API 支持
ToolStrategy 伪装成工具的 function calling，兼容性广
ReAct 循环 researcher → tools → researcher → … 直到满足退出条件

下一步

这周把图结构（三个 builder）和 state.py 基本看懂了，下一步要逐个理解节点函数：

  1. clarify_with_user — 学 with_structured_output 实际用法
  2. write_research_brief — 学如何把用户消息转成结构化任务
  3. supervisor — 学 bind_tools 实际用法
  4. supervisor_tools — 学工具调用解析 + 并发
  5. researcher + researcher_tools — 学完整的 ReAct 循环
  6. compress_research — 学如何处理上下文溢出

看每个函数前先问自己：这个函数从 State 读什么、写什么、在图里的位置是哪里。这个习惯帮助很大。

如果你也有 LLM 训练背景、想开始学 Agent 开发，open_deep_research 是个不错的切入点。代码结构清晰，把 LangGraph 的主要用法都覆盖了，踩一遍坑很值得。
